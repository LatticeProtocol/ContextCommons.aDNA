# Context Commons — Program Standing Orders

The five **procedural** standing orders (phase gates, destructive-action confirmation, context budget, local-context primacy, AAR-on-mission-close) live in the root `CLAUDE.md` because they govern session safety and apply to every agent run.

The seven **program-level** orders below govern Context Commons specifically — the curriculum, community relationships, pilot autonomy, youth-builder dignity, elder/steward authority, and the hardware/logistics realities that shape what can ship. They apply to any session that touches CC content, curriculum, pilot planning, or community relationships — load this file when domain context warrants.

## Program-level orders

1. **Archive, never delete.** Campaign docs, mission files, session records — permanent audit trail. Set `status: abandoned` or `status: completed`, never remove.
2. **Community voices are constitutional.** Every piece of content about a community must be reviewed for accuracy and representation by someone from that community. Never publish community content without confirming the community recognizes itself in it.
3. **The curriculum IS a context graph.** Curriculum modules are aDNA context files. When creating curriculum, you are also modeling what the students will learn to build. Self-referentiality is a feature, not a bug.
4. **Pilots are sovereign.** Each pilot community adapts the curriculum to its own context. Grand Rapids is not LA. Do not assume what works for one works for the other. Localization is a first-class design concern.
5. **Youth builders are peers, not students.** They are building real systems for real communities. Never condescend. The curriculum is rigorous; the relationship is collaborative.
6. **Elders and stewards have veto power on community use cases.** A technically elegant use case that the community stewards do not endorse does not ship. Community governance is a hard gate, not a soft signal.
7. **Hardware and logistics are first-class concerns.** Budget, internet connectivity, physical space, device availability — these are not implementation details. They are design constraints that shape curriculum and pilot design. Track them explicitly.

## When to load this file

- Any curriculum authoring, review, or revision session
- Any pilot-planning or pilot-evaluation work
- Any community-facing content draft (website pages, decks, video scripts) before community review
- Any session where someone proposes a use case for a pilot community
