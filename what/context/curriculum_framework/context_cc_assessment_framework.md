# Curriculum Framework: Assessment Framework

Evaluation methods aligned to tracks, milestones, and the capstone rubric. Assessment serves learning, not sorting.

## Assessment Philosophy

1. **Assessment serves learning.** Every evaluation moment is also a teaching moment. Formative feedback is continuous; summative gates are transparent. Assessment drives growth, not anxiety (Wiggins, 1998).
2. **Transparency.** Rubrics, criteria, and expectations are shared with participants before they begin. No hidden standards.
3. **Community validation is a hard gate.** A technically correct system that the community does not endorse does not pass (Standing Order #11). The end user's word is final on community value.
4. **Youth as peers.** Builders are assessed as collaborators, not test-takers (Standing Order #10). Code review and capstone evaluation are professional interactions, not school exams.
5. **Elder dignity.** Elders are never subjected to testing. Their participation is acknowledged, their contributions are traced through the artifacts they help create (Freire, 1970 — assessment as dialogue, not judgment).

## Assessment Methods by Track

| Method | Tracks | When | What It Measures |
|--------|--------|------|------------------|
| **Formative feedback** | All | Every session | Understanding, engagement, immediate course corrections |
| **Portfolio** | Builder, Steward | Accumulated over track | Growth artifacts: context graphs, reflections, assessments, plans |
| **Peer review** | Builder | Weeks 6-12 | Code and context quality assessed by fellow builders |
| **Capstone evaluation** | Builder | Week 12 | 4-criterion rubric: technical, governance, community, documentation |
| **Practicum evaluation** | Steward | Weeks 6-8 | Co-facilitation quality, needs assessment rigor, sustainability planning |
| **Community validation** | Builder, Steward | At milestones B2, B4, S3 | Steward/elder confirmation that work serves the community |
| **Self-assessment** | Elder | Ongoing | Self-reported confidence, engagement, satisfaction |

## Capstone Evaluation (Builder Track, B4)

The capstone is the program's summative assessment. Three evaluators review against four criteria:

| Criterion | Evaluator | Developing | Proficient | Exemplary |
|-----------|-----------|-----------|-----------|-----------|
| **Technical quality** | Lattice mentor | Happy path works; structural gaps in context graph | Common cases handled; graph well-structured | Edge cases handled; graph clean, efficient, maintainable |
| **Governance compliance** | Community steward | Mostly correct data handling; minor consent gaps | Full compliance with community rules and consent | Proactive governance; builder identifies and resolves edge cases |
| **Community value** | End user | Usable with assistance | Independently useful | Actively requested and recommended by user |
| **Documentation** | All three | Context graph exists but hard to follow | Legible; another builder could maintain it | Model for others; complete triad, clear annotations |

**Pass condition**: All three evaluators rate Proficient or above on all four criteria. Disagreements mediated by steward. A technically exemplary system the end user doesn't find useful does not pass.

**Iteration, not failure**: Builders whose capstone does not pass on first evaluation receive specific, actionable feedback and a revision window. The goal is deployment, not gatekeeping. Builders retain shared workstation access until their capstone passes.

## Community Validation Protocol

Community validation is not a form or a survey. It is a structured conversation:

1. **Setting**: Private, comfortable, in the participant's preferred language. Steward facilitates.
2. **Questions** (adapted to context):
   - "Does this system help you with [the identified need]?"
   - "Would you continue using it?"
   - "Is there anything it does that makes you uncomfortable?"
   - "Would you recommend it to [family / neighbors / congregation]?"
3. **The community member's word is final.** If the elder says "this doesn't help me," the system does not pass community validation regardless of technical quality.
4. **Dignity is paramount.** The elder is not being tested. They are being consulted. The conversation honors their expertise as the person who knows their own needs best.

## Localization of Assessment

Assessment criteria are universal — the 4-criterion capstone rubric does not change across pilot sites. Assessment *methods* may be adapted:

| Adaptation | When | Example |
|------------|------|---------|
| Oral demonstration | Low-literacy participants | Builder explains capstone verbally instead of written documentation |
| Multilingual submission | Non-English-dominant participants | Portfolio artifacts in participant's primary language with English summary |
| Extended timelines | Accessibility needs | Additional revision window for builders with learning disabilities |
| Alternative evidence | Community context | Steward video-records needs assessment instead of written document |

The Curriculum Council approves adaptations. Localization changes the *how* of assessment, never the *what* — community value, technical quality, governance compliance, and documentation are always evaluated.
